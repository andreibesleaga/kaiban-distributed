# Developer Workflow & Documentation

> **Source Origin**: Distilled from Matteo Collina's workflow skills (`octocat`, `documentation`, `snipgrapher`).

This guide standardizes how GABBE agents execute Git operations and format project documentation.

---

## 1. Documentation Framework (Diátaxis)

When researching, authoring, or rewriting documentation, strictly adhere to the **Diátaxis** framework, which divides technical documentation into four distinct quadrants. Never mix these concerns in the same file without clear separation.

1.  **Tutorials** (Learning-oriented):
    -   *Goal*: Allow the newcomer to achieve a basic outcome.
    -   *Style*: Step-by-step, pedagogical, conversational. Do not explain *why*, just show *how*.
2.  **How-To Guides** (Problem-oriented):
    -   *Goal*: Help the user solve a specific real-world problem.
    -   *Style*: Recipe-like, practical, focused. Assume the user already knows the basics.
3.  **Reference** (Information-oriented):
    -   *Goal*: Accurately describe the machinery (API specs, configs, CLI flags).
    -   *Style*: Austere, complete, structured. No explanations or tutorials—just facts.
4.  **Explanation** (Understanding-oriented):
    -   *Goal*: Clarify architecture, design decisions, and underlying concepts.
    -   *Style*: Discursive, theoretical, big-picture. (e.g., "Why we chose Fastify over Express").

## 2. Advanced Git & GitHub Operations (Octocat)

Agents performing version control must operate as Git/GitHub wizards using both local git and the `gh` CLI.

-   **Commit Hygiene**: Use Conventional Commits (`feat(scope): message`, `fix(api): message`). Each commit must represent a single logical unit.
-   **Branching Strategy**: Use short-lived feature branches (`feat/name`, `fix/name`, `docs/name`). Never commit directly to `main` without merging via PR.
-   **Rebasing over Merging**: Keep linear histories. Prefer `git pull --rebase` and interactive rebasing to squash fixup commits before opening a PR.
-   **GitHub CLI**: Leverage `gh` for PR creation (`gh pr create`), status checks (`gh pr checks`), and issue tracking (`gh issue list`).

## 3. Visual Aids & Snippets (Snipgrapher)

When preparing documentation intended for presentation or social reading (like blog posts or internal showcase PRs):
-   Use polished code snippet graphics (like those generated by `snipgrapher`).
-   Ensure syntax highlighting perfectly matches the provided code.
-   Maintain visual contrast and accessibility.
-   Keep the code snippet concise and focused strictly on the subject matter being highlighted.
